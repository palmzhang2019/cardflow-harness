# Requirement Card Candidate

## 1. 原始需求保真

```text
"【① 顧客情報レポートについて】
「顧客最新情報」においても発信日付を表示できるようしたい
※現状は「顧客対応履歴」から全履歴を確認いただき、最新の発信日付を抽出いただく必要がございます。"

ベルシステム２４
```

用户补充原话 / 线索：

```text
発着信日時（最新）→ 発着信履歴に一番最新→ cdr参照

* 表形式レポート
* 集計レポート
```

---

## 2. 一句话需求摘要

`admin/customer_contact_history_reports/add` ページの「項目（*必須）」に「発着信日時（最新）」という選択肢を追加し、レポート上で各顧客の最新発着信履歴の日時を表示できるようにする。

---

## 3. 用户真实目标

- レポートを作成・閲覧する担当者が、顧客ごとの最新の発着信日時を一目で確認できるようにする。
- これにより、現在「顧客対応履歴」画面から手作業で最新日時を探している手間を省く。

---

## 4. 业务对象与操作路径

- **対象レポート種別**: 顧客情報レポート（コード上では `target=2`）
- **対象画面**:
  - レポート新規作成: `admin/customer_contact_history_reports/add`
  - レポート編集: `admin/customer_contact_history_reports/edit`
  - レポート表示: 表形式レポート詳細画面
- **操作の流れ**:
  1. ユーザーがレポート作成画面を開く
  2. 「項目（*必須）」ブロックに表示される選択肢一覧から「発着信日時（最新）」を探す
  3. チェックボックスを ON にする
  4. 「順番」は一覧の最後尾に自動設定されていることを確認する
  5. レポートを保存し、表形式で表示する
  6. 表示されたレポートの列に「発着信日時（最新）」が含まれ、各顧客の最新発着信日時が表示される

---

| 情報取得手段 | ユーザーは「顧客対応履歴」画面から全履歴を確認し、目視で最新の発信日付を抽出している。 |

## 6. 期望结果

- `admin/customer_contact_history_reports/add` 画面の「項目（*必須）」に、`出力 項目名` 列と `表示名` 列に「発着信日時（最新）」が表示される。
- `順番` は当該ブロック内の最後尾に位置する。
- ユーザーがこの項目をチェックしてレポートを保存できる。
- 保存後、表形式レポートの表示画面およびダウンロード（CSV）において、顧客ごとに「発着信日時（最新）」の値が表示される。
- 表示される値は、**当該顧客の発着信履歴の中で最も新しい「発着信時間」** であること。

---

## 7. 明确范围 In Scope

- **対象レポートタイプ**: 表形式レポート（`type=1`）
- **対象集計対象**: `target=2`（顧客最新情報）
- **追加項目**:
  - システムフィールドとして `発着信日時（最新）` を定義
  - 画面（add / edit）の「項目（*必須）」一覧に当該項目を追加
  - 順番はシステムフィールド一覧の最後尾とする
- **レポート出力対応**:
  - 表形式レポートの画面表示（DataTables）
  - 表形式レポートの CSV ダウンロード
- **データ取得ロジック**:
  - 顧客ごとに発着信履歴（cdr）の `start_date` が最大となるレコードを参照し、その `start_date` を表示する（※口径は要確認、後述）
- **表示文言**:
  - 日本語ロケールのプロパティファイルに `発着信日時（最新）` を追加
- **回帰影響確認**:
  - 既存の「項目（*必須）」の他項目が影響を受けないこと
  - 当該項目を未チェックの場合、既存レポート出力内容が変わらないこと

## 9. 需要确认 Unknowns

以下の事項は、本 M1 段階では **未確定** であり、後続工程での回答が必要です。

1. **レポート種別の範囲**  
   - ユーザー補足に「集計レポート」とあるが、今回のスコープは **表形式レポートのみ** か、それとも **集計レポートでも選択可能にする** 必要があるか。  
   - コード上、集計レポートは「項目（*必須）」とは別の X/Y/V 選択方式であるため、集計を含める場合は対応方式が大きく異なる。

2. **「最新」の定義**  
   - `発着信日時（最新）` の“最新”は、**Cdr.start_date の最大値** と定義してよいか。  
   - または、現在 `target=2` で使われている「顧客最新情報」のロジック（`customer_notes` の最大 `id` で最新ノートを特定し、それに紐づく cdr を参照）に合わせる必要があるか。

3. **無データ時の表示**  
   - 顧客に発着信履歴が存在しない場合、または該当する cdr レコードが取得できない場合、**何を表示すべきか**（空文字、`-`、`（空白）`、`N/A` など）。

4. **順番の「最後位」の具体的意味**  
   - 「順番」を最後尾に置くとは、**システムフィールド一覧の末尾** を指すか。  
   - あるいは、**システムフィールド＋カスタムフィールド（`CustomerContactExt`）を含む全項目リストの末尾** を指すか。  
   - 現状、カスタム項目はシステム項目の後ろに追加されるため、どちらに置くかで画面表示順が変わる。

5. **CSV ダウンロードの扱い**  
   - 表形式レポートの CSV ダウンロードは、**本件のスコープに必ず含める** 必要があるか。  
   - それとも、画面表示のみを優先し、CSV は別タスクとするか。

6. **API `api_find_report` への影響**  
   - 外部連携などで使用される `api_find_report` エンドポイントにも、同様に `発着信日時（最新）` の項目を追加する必要があるか。  
   - 特に、`target=2` のレポートが API 経由で呼ばれるケースを想定する必要があるか。

7. **`顧客最新情報` における「発信日付」と「発着信日時（最新）」の名称統一**  
   - リーダー原話では「発信日付」、ユーザー補足は「発着信日時（最新）」。最終的にシステム上で表示する文言は **発着信日時（最新）** で確定か。それとも「発信日付」と表記するか。

## 10. 验收标准 Acceptance Criteria

### AC-1: 項目表示
- **環境**: 本番同等環境にデプロイ後
- **操作**: 管理画面から `admin/customer_contact_history_reports/add` を開き、`campaign` と `target`（`顧客最新情報` に相当するもの）を選択する
- **期待結果**: 「項目（*必須）」の一覧に **`発着信日時（最新）`** という行が表示され、`出力 項目名` 列と `表示名` 列に当該文言が確認できること

### AC-2: 順番
- **操作**: AC-1 と同じ画面で「項目（*必須）」一覧を確認する
- **期待結果**: `発着信日時（最新）` の `順番` が、当該ブロック内の **最後尾の番号** であること（システムフィールドの最終行として表示されること）

### AC-3: チェックと保存
- **操作**: `発着信日時（最新）` のチェックボックスを ON にし、その他必須項目を入力してレポートを保存する
- **期待結果**: エラーなく保存され、編集画面で再度開いた際にチェック状態が維持されていること

### AC-4: レポート出力（表形式・画面）
- **操作**: 保存したレポートの「表形式」表示画面を開く
- **期待結果**: 表示されるテーブルの列に `発着信日時（最新）` が存在し、各顧客の行に **最新の発着信日時**（例: `2026-05-13 10:30:00`）が表示されること

### AC-5: レポート出力（最新判定）
- **操作**: テスト用に、同一顧客に対して複数の発着信履歴（cdr）を異なる日時で用意し、レポートを表示する
- **期待結果**: レポート上の `発着信日時（最新）` 列には、**最も新しい `start_date` の値** が表示されること（※口径が確定した定義に従う）

### AC-6: 無データ時の表示
- **操作**: 発着信履歴を１件も持たない顧客を含む状態でレポートを表示する
- **期待結果**: 当該顧客の `発着信日時（最新）` 列には、**事前に合意された無データ表示**（空文字 または `-` など）が表示されること

### AC-7: 回帰確認（項目非選択時）
- **操作**: `発着信日時（最新）` を **チェックせずに** 保存したレポート、および既存のレポートを開く
- **期待結果**: 既存の項目（`顧客ID`、`顧客名` など）が従来通り表示され、新たな列は追加されず、並び順も変わらないこと

### AC-8: CSV ダウンロード（スコープに含む場合）
- **操作**: `発着信日時（最新）` を含むレポートの「CSVダウンロード」を実行する
- **期待結果**: CSV のヘッダに `発着信日時（最新）` が含まれ、データ行には画面表示と同じ値が出力されること

## 11. 风险与歧义

| # | 风险 / 歧义 | 影響 | 深刻度 |
|---|-------------|------|--------|
| R1 | **「最新」の口径不一致**：現状の `target=2` は `customer_notes.max(id)` で最新ノートを特定し、そこから cdr を参照する仕様。一方、直感的には cdr.start_date の最大値を期待する可能性が高い。 | ビジネス要件と実装がずれると、表示される日時が“最新発着信”でなく“最新ノートの発着信”になる。出荷後に手戻りが発生する。 | 高 |
| R2 | **集計レポートの範囲未確定**：ユーザー補足に「集計レポート」の言及があるが、技術的には全く別構造。スコープに含める場合、追加の設計・実装コストが大きい。 | 最初から集計まで求められると、見積もり・期間が大きく変わる。後から追加依頼が来る可能性もある。 | 中 |
| R3 | **無データ表示の不統一**：複数のパスで空文字／`（空白）`／`N/A` などが混在しており、今回の要件で統一した表示ルールが決まっていない。 | UI 上の一貫性が損なわれ、ユーザーが混乱する。 | 低 |
| R4 | **順番の解釈揺れ**：カスタム項目を含む全リストの最後に置くべきか、システム項目だけの最後に置くべきかで、実際の表示位置が変わる。 | リリース後に「思っていた最後と違う」という指摘を受ける可能性がある。 | 低 |
| R5 | **CSV / API 影響の漏れ**：画面表示のみを想定して進めると、ダウンロードや外部連携で項目が欠落し、業務フローが破綻する。 | 重要な導出機能が使えず、問い合わせや障害扱いになる。 | 中 |
| R6 | **既存レポートの互換性**：既存レポートの設定データに新しいフィールド定義が混ざることで、予期しない表示崩れやエラーが発生する可能性がある。 | 全レポートの動作確認が必要になる。 | 中 |

## 12. 给后续 Engineering Requirement Card 的输入提醒

1. **最新判定ロジックの確定**  
   必ず `Cdr.start_date MAX` で取得するのか、現行の `customer_notes.id MAX` 経由のロジックを踏襲するのかを決定すること。後者の場合、項目名を「最新ノート発着信日時」などに変更する必要性も検討する。

2. **集計レポートの要否**  
   ユーザー補足から漏れなく拾うため、ステークホルダーに **集計レポートでも選択可能にする必要があるか** を再確認する。必要と判断された場合は、Engineering Requirement Card を分割し、集計レポート用の追加タスクとして定義する。

3. **無データ時の統一表示**  
   システム全体の方針または本レポート固有のルールとして、値が取得できない場合の表示（例: `-`）を決定し、全出力経路（画面、CSV、将来的なAPI連携）に適用する。

4. **順番の配置ルール**  
   システムフィールドの末尾に置くのか、カスタムフィールドを含めた末尾に置くのかを決定し、実装時に `num` の割り振り方を指示する。

5. **CSV および API の対応範囲**  
   少なくとも表形式レポートの CSV ダウンロードは同一スコープに含めることを推奨する。`api_find_report` については、利用実態を確認のうえ、必要に応じて子タスクを切る。

6. **多言語リソースの準備**  
   日本語のプロパティ `ApplicationResources.properties` に加え、他言語（英語、中国語など）のリソースファイルにも同様のキーを定義する必要があるか否かを確認する。

7. **既存データのマイグレーション**  
   既存のレポート設定に新しいフィールドが自動的に現れないよう、追加は新規作成/編集時のみ影響する設計とする。必要に応じて、既存レポートの「項目」一覧に当該フィールドが追加されないことを保証する。

8. **権限制御**  
   `privilege.conf` にレポートダウンロード用の未使用アクションが存在するため、今回の対応で混入させないよう注意する。新たな権限制御は不要だが、念のため Engineering Requirement Card に「権限変更なし」と明記する。

9. **テストデータの準備**  
   最新発着信の判定を検証するため、同一顧客に `start_date` の異なる複数の cdr を持つテストデータが必要となることを Engineering Requirement Card に記載しておく。
