## 3. Selection Reason

1. **评分 winner**：08-score.json 中 Claude 总分 80（满分 80），且在 8 个评分维度全部达到满分；Codex 76、DeepSeek 55，差距明确，无反转理由。
2. **二分结构最稳健**：Claude 第 6 节显式分离「展示项目层面（已确认）」与「最新判定口径层面（必须保留为 Unknown）」，直接、显眼地隔离了本任务最高优先级的口径风险，是三张卡中最难被下游 Engineering Requirement Card 误读为「最新 = `cdr.start_date` 最大值」的结构。
3. **AC 扩展挂载点清晰**：Claude 第 10 节专门分出「依赖 Unknown 的待补 AC」段，把集計 / CSV / API / 权限 / 多语言全部挂载在 Unknowns 上，frozen 后扩展路径无歧义。
4. **零实现泄露**：Claude 通篇没有出现 `customer_notes.id 最大值` 的具体引用（仅以「现有最新链路」抽象表述），没有 `id DESC` 等 ORDER BY 级实现细节，没有具体字段命名 / SQL / i18n key。
5. **Unknowns 与风险覆盖面与 Codex 持平**：Unknowns 14 项 / 风险 13 项，与 Codex 同等覆盖面但更紧凑、无重复，且权限可见性、多语言、列宽 / 排序 / 筛选等表现层条目齐备（DeepSeek 在这些维度有遗漏）。
6. **元约束置顶**：Claude 第 9 节末尾「以上 Unknowns 必须保留，禁止在本卡片中替领导脑补回答」对 frozen 后下游卡片的约束力最强。
