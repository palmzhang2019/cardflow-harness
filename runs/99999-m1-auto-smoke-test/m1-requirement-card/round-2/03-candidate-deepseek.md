# Requirement Card Candidate

## 1. 原始需求保真

领导原话（保留原文）：

```text
"【① 顧客情報レポートについて】
「顧客最新情報」においても発信日付を表示できるようしたい
※現状は「顧客対応履歴」から全履歴を確認いただき、最新の発信日付を抽出いただく必要がございます。"

ベルシステム２４
```

补充原话 / 线索（保留原文）：

```text
発着信日時（最新）→ 発着信履歴に一番最新→ cdr参照

* 表形式レポート
* 集計レポート
```

**已确认**  
- 目标页面是 `admin/customer_contact_history_reports/add`  
- 需求围绕 `顧客情報レポート` / `顧客最新情報`  
- 希望新增一个可选项目 `発着信日時（最新）`  
- 现状下，用户需要从 `顧客対応履歴` 全履历中人工确认最新发信日付  

**推测**  
- 领导原话中的“発信日付”在实际页面/报表项上可能落为 `発着信日時（最新）`  
- 数据来源可能与 `cdr` 相关  

**未知**  
- 领导原话中的“発信日付”与补充线索中的“発着信日時（最新）”是否完全同义  
- 本次验收是否覆盖表形式、集計、CSV、API 等全部相关出口

## 2. 一句话需求摘要

在 `admin/customer_contact_history_reports/add` 页面中，为 `顧客最新情報` 对应报表新增可选项目 `発着信日時（最新）`，以减少用户手工翻查履历的成本；但该项目所展示的“最新”判定方法以及具体影响范围（表形式 / 集計 / CSV / API 等）尚未确认。

## 3. 用户真实目标

**已确认**  
- 降低用户为查看最新发信相关时间而手工翻查 `顧客対応履歴` 的操作成本  
- 让 `顧客情報レポート` 中的 `顧客最新情報` 直接承载该信息  

**推测**  
- 用户希望在报表配置阶段勾选该项后，生成结果时直接看到对应值  
- 用户希望该字段的行为与现有报表字段机制保持一致  

**未知**  
- 用户真实目标是否仅限于“看见这个字段”，还是同时期待该字段参与集計、CSV、API 或其他导出 / 对外输出  
- 是否存在特定角色下该新增项目不可见或不可选的业务约束

## 5. 当前现状

**已确认**  
- `admin/customer_contact_history_reports/add` 的 `項目（*必須）` 由既存字段定义生成  
- 当前 `顧客最新情報` 对应字段列表中没有 `発着信日時（最新）`  
- 当前报表输出依赖已保存的字段配置进行展示  
- 当前 `顧客最新情報` 链路中的“最新信息”现有实现口径与 `customer_notes` 的最新记录（最大 `id`）有关  
- `/admin/cdr/index` 中存在 `発着信時間`，其数据来源为 `cdr.start_date`  

**推测**  
- 现状下用户无法在 `顧客最新情報` 报表中直接看到所需的最新発着信日時  
- 现有“最新 note”口径与“最新 cdr”口径可能不一致  
- 集計レポートのデータ取得構造が表形式とは異なり、同フィールドを追加する場合には別途検討が必要になる可能性がある  

**未知**  
- 无 `発着信履歴` 数据时，现有同类字段的期望展示方式  
- 当前集計报表在业务上是否有承载该字段的合理性  
- `api_find_report` 当前是否被业务方实际作为验收对象使用

## 10. 验收标准 Acceptance Criteria

**正向项**  
- AC-001：`admin/customer_contact_history_reports/add` に手動でアクセスし、`項目（*必須）` 内に `発着信日時（最新）` が表示されること  
- AC-002：`発着信日時（最新）` の `出力 項目名` 列の値が `発着信日時（最新）` であること  
- AC-003：`発着信日時（最新）` の `表示名` 列の値が `発着信日時（最新）` であること  
- AC-004：`発着信日時（最新）` の `順番` が最後尾に表示されること（具体的な基準は Unknowns #5 の確認結果に従う）  
- AC-005：`発着信日時（最新）` を手動でチェックできること  
- AC-006：設定保存後、再度編集／表示画面に入った際に `発着信日時（最新）` のチェック状態が維持されていること  
- AC-007：`発着信日時（最新）` をチェックした状態でレポートを生成／表示し、レポート上に `発着信日時（最新）` 列が存在すること  
- AC-008：複数の発着信履歴を持つ顧客を用意し、レポートに表示される当該顧客の `発着信日時（最新）` の値が、業務上取り決めた“最新”の日時と一致すること（“最新”の定義は Unknowns #3 の確認結果に従う）  
- AC-009：発着信履歴が存在しない顧客について、当該列の表示が Unknowns #4 の確認結果に従った表示であること  

**回归项**  
- AC-Regression-001：`発着信日時（最新）` をチェックしない場合、既存レポートの項目表示が従来通りであること  
- AC-Regression-002：`項目（*必須）` 内の既存項目の表示名、順番、チェック動作が本対応によって意図せず変化しないこと  
- AC-Regression-003：`発着信日時（最新）` をチェックしない状態、または `Cdr.start_date` が関係しない他のシナリオにおいて、既存の `Cdr.start_date` 関連挙動が影響を受けないこと  

**范围依赖 AC（确认后补充）**  
- 集計レポートに関する AC：Unknowns #2 の確認結果を待って補足する  
- CSV（表形式 / 集計）に関する AC：Unknowns #6 の確認結果を待って補足する  
- `api_find_report` に関する AC：Unknowns #7 の確認結果を待って補足する  

> 今回の受入範囲が集計／CSV／API を含まないと確定した場合、当該出口は合格判定から除外する。

## 11. 风险与歧义

| # | 风险 / 歧义 | 影响 / 深刻度 |
|---|-------------|---------------|
| 1 | `顧客最新情報` の現行“最新”実装（`customer_notes.id` 最大）と、ユーザーが想定する“cdr 参照”による最新発着信日時の定義が食い違う可能性 | 生成される値が業務期待と異なり、リリース後の手戻りを招く（高） |
| 2 | `表形式レポート` と `集計レポート` でフィールド追加の仕組みが異なり、範囲未確定のまま進行すると需要の拡大や仕様の不一致が生じる | 範囲・工数・受入基準が変動する（高） |
| 3 | `/admin/cdr/index` のデフォルト表示順が `id DESC` であり、厳密な `start_date DESC` とは一致しない。この画面の“見た目の最新”と最大値が同一でない場合、業務基準との乖離を生む | 暗黙の前提により誤った口径が設計に持ち込まれる（高） |
| 4 | CSV（表形式／集計）、`api_find_report` への波及が未確定 | 受入基準の分岐、重要な出力経路の見落とし（中） |
| 5 | リーダー発言「発信日付」と補足「発着信日時（最新）」の表記揺れ、および「発信」と「発着信」の意味範囲の曖昧さ | 項目名・表示ラベルの不一致、データ抽出範囲の解釈違い（中） |
| 6 | 「日付」と「日時」の粒度の違い（日単位か時刻単位か） | 出力フォーマット・表示精度が期待と異なる（中） |
| 7 | コード調査は「現在どう実装されているか」を示すに過ぎず、業務上の正解ではない | 調査結果をそのまま業務要件と誤認すると、要求と異なる設計になる（中） |
| 8 | 「順番 最后一位」の基準（システムフィールド末尾／全リスト末尾）が未確定 | 実際の表示位置が期待と異なる（低） |
| 9 | 無データ時の空値表示ルールが未確定 | 複数の出力先で表示が不統一になり、受入時に混乱を招く（低） |
| 10 | M1 段階で Unknown を事実化してしまうと、後続の Engineering Requirement Card に誤った前提が伝播する | 範囲・ロジックの誤りが下流工程に拡大する（高） |

## 12. 给后续 Engineering Requirement Card 的输入提醒

Engineering Requirement Card 着手前に、少なくとも以下を確定させること：  
- 本件の対象範囲が `表形式レポート` に限定されるか否か  
- `集計レポート` を対象に含めるか  
- CSV（表形式／集計）、`api_find_report` への波及要否  
- `発着信日時（最新）` の“最新”を決定する業務ロジック  
- 発着信履歴が存在しない場合の表示ルール  
- `順番` の“最后一位”の判定基準  
- 最終的な表示名称（「発信日付」か「発着信日時（最新）」か）  

また、以下の点を Engineering Requirement Card へ明示的に引き継ぐこと：  
- 本カードで保持しているすべての Unknowns およびリスクは、Engineering Requirement Card 内で「確認／推迟／範囲外」のいずれかに処置を明示すること  
- コード調査報告書の「関連可能性のあるファイル」「最小変更候補範囲」「呼び出しチェーン」は、あくまで後続工程での参考情報であり、M1 段階で「修正必須ファイル」や実装指示に格上げしてはならない  
- 現時点で確認できている現状の実装ロジック（`顧客最新情報` における最新 note 参照、`cdr` テーブルとの結合状況など）は「現在のシステムの振る舞い」であり、必ずしも今回の正解ではない  

**テストデータ準備に関する注意**  
- 最新発着信判定の検証のため、同一顧客に対して `start_date` の異なる複数件の `cdr` レコードを用意したテストデータが必要となる可能性が高い
