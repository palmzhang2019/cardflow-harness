# Round 2 Candidate Comparison

## 1. 总体结论

三张 Candidate 整体水平都很高，均严格保留了领导原话、严格区分了已确认 / 推测 / 未知，没有把核心 Unknown（"最新 = Cdr.start_date 最大值"）写成确定事实，没有进入实现方案、SQL 或 patch。三者都识别出了表形式 / 集計 / CSV / API 的范围未知，并将其作为 Unknown 保留。

差异主要体现在：

- **Claude (B)** 在第 6 节明确把「展示项目」与「最新判定口径」做了显式分离，并在 AC 中专门列出「依赖 Unknown 的待补 AC」段落，结构最严谨，最适合作为 frozen 主卡的基础。
- **Codex (A)** 与 Claude 几乎同等水平，Unknowns 与风险条目最齐全（14 / 13 条），表达稍冗长，但完备性最高。
- **DeepSeek (C)** 整体合格，但在「当前现状」与「风险」段轻度泄露了代码调查的实现细节（`customer_notes.id 最大值`、`/admin/cdr/index 的デフォルト表示順が id DESC`），且 Unknowns（11 条）与风险（10 条）的覆盖面相对偏窄，缺少多语言文案 / 列宽 / 排序 / 筛选等表现层条目。

## 2. Candidate A: Codex 评价

### 优点
- 领导原话与补充线索完整保留，未改写。
- Unknowns 14 条、风险 13 条，覆盖面最广，包含「角色级权限可见性」「多语言」「列宽 / 排序 / 筛选」「`/admin/cdr/index` 视觉最新 ≠ 严格最新」等隐性风险。
- 严格区分已确认 / 推测 / 未知，没有把任何 Unknown 转写为事实。
- 「最新」口径明确作为 Unknown #3 保留，未把 `Cdr.start_date 最大值` 当成既定规则。
- 集計レポート、CSV、表形式 CSV、集計 CSV、`api_find_report` 全部作为待确认范围分别列出。
- AC 全部可手动验证，并显式声明依赖 Unknown 的 AC（AC-010 ~ AC-012）以 Unknowns 确认结果为准。
- 第 12 节给后续 Engineering Requirement Card 的提醒最完整，明确禁止把「可能相关文件」升级为「必须修改文件」。

### 问题
- 第 5 节「当前现状」中「当前 `顧客最新情報` 链路中的"最新信息"现有实现口径与最新 note 有关」属于代码调查结论的轻度引入；虽然作为「现状」描述尚可接受，但比 Claude 的表述略具体。
- 整体偏长，部分条目与第 11 节风险存在轻度重复（如 Unknowns #9 / 风险 #4 / 风险 #5 / 风险 #6 之间有交叉）。
- 一句话需求摘要中「但该字段"最新"的判定口径及影响范围仍待确认」与第 6 节有重复表述。

### 可吸收内容
- 第 12 节「测试数据准备提醒」中 `start_date` 不同的多条记录测试数据要求。
- 风险表第 13 条「M1 阶段提前事实化 Unknown 会污染 Engineering Requirement Card」的元约束。
- 第 12 节「代码调查中已确认的是『当前如何实现』，不是『本次必须如何实现』」的引导语。

## 3. Candidate B: Claude 评价

### 优点
- 领导原话与补充线索完整保留。
- **第 6 节「期望结果」明确分离「展示项目层面（已确认）」与「最新判定口径层面（必须保留为 Unknown）」**，是三张 Candidate 中边界最清晰的处理，最直接对应「最新口径不得提前事实化」这一关键风险。
- AC 段专门列出「依赖 Unknown 的待补 AC」（集計 / CSV / API / 权限可见性 / 多语言），结构最清晰，frozen 后扩展路径明确。
- Unknowns 14 条，风险 13 条，覆盖面与 Codex 相当。
- 集計レポート与表形式レポート的承载机制差异被显式标注。
- 权限可见性作为 Unknown #14 保留，并在第 11 节风险 #12 与「本次不主动改权限控制」的推测之间显式标出「解释空间」。
- 完全没有混入实现方案。

### 问题
- 与 Codex 类似，第 5 节也引用了「`customer_notes.max(id)` 链路」这一现有实现表述（在第 6 节 Unknowns 中），属于代码调查结论的轻度引入；但被严格限定为「现有口径」而非「本次口径」，问题较轻。
- 第 11 节风险 #9 表述「视觉最新 ≠ 严格最新」相对抽象，未明确指出基准差异的具体形式（这点 DeepSeek 反而更具体，但 DeepSeek 也因此泄露了实现细节，属于权衡）。

### 可吸收内容
- 第 6 节「展示项目」与「最新判定口径」的显式二分结构 —— 这是最值得吸收到主卡的设计。
- 第 10 节「依赖 Unknown 的待补 AC」分段写法。
- 第 11 节风险 #9「视觉最新 ≠ 严格最新」的隐性偏差概念。

## 4. Candidate C: DeepSeek 评价

### 优点
- 领导原话与补充线索完整保留。
- 第 6 节末尾显式注明「本节将『表示する項目』と『値を決定するロジック』を分離して記述」，与 Claude 的二分思想一致。
- 第 11 节风险 #3 指出了 `/admin/cdr/index` 默认表示順 `id DESC` 与严格 `start_date DESC` 不一致的隐性偏差，比 Codex / Claude 更具体（但同时也是一个问题，见下）。
- 用日语 / 中日混排表述 Unknowns 与 AC，对原始业务方语境保真度较高。
- 集計レポート与表形式レポート的差异显式标注。

### 问题
- **轻度混入了代码调查的实现细节**：
  - 第 5 节「当前现状」中「当前 `顧客最新情報` 链路中的"最新信息"现有实现口径与 `customer_notes` 的最新记录（最大 `id`）有关」—— 比 Codex / Claude 的同类描述更具体，已经接近实现细节。
  - 第 5 节「`/admin/cdr/index` 中存在 `発着信時間`，其数据来源为 `cdr.start_date`」—— 「数据来源为 `cdr.start_date`」属于代码调查结论的显式引入。
  - 第 11 节风险 #3「`/admin/cdr/index` 的デフォルト表示順が `id DESC`」—— 进入了 ORDER BY 级别的实现细节。
- **Unknowns 数量偏少（11 条）**：缺少「多语言文案 / 翻译范围」「列宽 / 排序 / 筛选默认行为」「表形式 CSV / 集計 CSV 是否分别属于范围」等条目（CSV 仅笼统提及一次）。
- **风险数量偏少（10 条）**：缺少「代码调查不能替代业务确认」级别的元风险；缺少「列宽 / 排序 / 筛选」表现层风险。
- 第 12 节虽然提到「代码调查報告書」的引用约束，但比 Codex / Claude 略简短。
- 没有把「最新 = `Cdr.start_date` 最大值」当成事实，这一点合格；但风险 #3 的表述方式（直接引用 `id DESC` 排序基准）已经为下游工程卡引入了潜在的实现倾向。

### 可吸收内容
- 第 6 节关于「分離して記述」的元说明。
- 风险 #3 的核心思想（视觉最新 ≠ 严格最新），但表述应抽象化，去除 `id DESC` 这种实现细节。

## 7. 推荐主卡

**推荐 Candidate B: Claude 作为后续 frozen requirement card 的主卡基础。**

核心理由：

1. **第 6 节对「展示项目」与「最新判定口径」的显式二分**，直接、显眼地隔离了本任务最高优先级的口径风险，是三张卡中最难被下游工程卡误读的结构。
2. **第 10 节 AC 中显式分出「依赖 Unknown 的待补 AC」**，让 frozen 后的扩展路径（集計 / CSV / API / 权限 / 多语言）有清晰挂载点。
3. Unknowns 与风险的覆盖面与 Codex 持平，但行文比 Codex 紧凑、重复更少。
4. 没有像 DeepSeek 那样混入 `customer_notes.id 最大值` / `id DESC` 等代码调查实现细节。

Codex 是非常接近的候选，可作为副参考；DeepSeek 在被剔除实现细节、补足 Unknowns 后，部分条目可作为风险描述的二级来源。

## 8. 合并建议

以 **Claude (B)** 为主卡骨架，对 Codex (A) 与 DeepSeek (C) 中可吸收的内容做定向合并：

1. **保留 Claude 的整体结构与第 6 节二分**（展示项目 / 最新判定口径），不要改写为单一段落。
2. **从 Codex 吸收**：
   - 第 12 节末尾的「测试数据准备提醒」（同一顧客に `start_date` の異なる複数件の発着信履歴データを準備）。
   - 风险表关于「代码调查中已确认的是『当前如何实现』，不是『本次必须如何实现』」的引导语，作为元约束置于第 12 节。
   - Unknowns 中若 Claude 有遗漏的细分项（如「表形式 CSV / 集計 CSV 是否分别属于范围」）需对齐。
3. **从 DeepSeek 吸收（需改写）**：
   - 第 11 节风险 #3「`/admin/cdr/index` 视觉最新 ≠ 严格最新」的具体场景描述，**但必须删除 `id DESC` 这种 ORDER BY 级实现细节**，改为抽象的「该画面的默认排序基准与严格的『最新』口径可能不一致」（Claude 风险 #9 已是这种抽象形式，可直接保留 Claude 原文）。
4. **明确不吸收**：
   - DeepSeek 第 5 节「数据来源为 `cdr.start_date`」「`customer_notes` 的最新记录（最大 `id`）」等接近实现的现状描述 —— 保留 Claude 与 Codex 的抽象级表述即可。
5. **统一表述**：将合并卡中所有「最新」相关 AC 与 Unknown 全部加注「以 Unknowns #3 确认结果为准」尾注，确保 frozen 后下游工程卡无法在未确认前消解此项。
6. **保持 Unknowns 与风险的元约束置顶**：「以上 Unknowns 必须保留，禁止在本卡片中替领导脑补回答」这句必须随 Claude 主卡一起进入 frozen 版本。
