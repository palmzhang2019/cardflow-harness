# Requirement Card Candidate

## 1. 原始需求保真

领导原话：

```text
"【① 顧客情報レポートについて】
「顧客最新情報」においても発信日付を表示できるようしたい
※現状は「顧客対応履歴」から全履歴を確認いただき、最新の発信日付を抽出いただく必要がございます。"

ベルシステム２４
```

用户补充原话 / 线索：

```text
発着信日時（最新）→ 発着信履歴に一番最新→ cdr参照

* 表形式レポート
* 集計レポート
```

## 2. 一句话需求摘要

在 `admin/customer_contact_history_reports/add` 页面、当対象为「顧客最新情報」时，于 `項目（*必須）` 中追加一个可选项目 `発着信日時（最新）`，让报表可直接展示该项目；但该字段中“最新”的判定口径仍待业务确认。

## 3. 用户真实目标

- 已确认：让使用「顧客最新情報」レポート的用户，不必再回到「顧客対応履歴」逐条翻看后人工提取日期。
- 已确认：希望在当前报表使用链路中，直接看到一个名为 `発着信日時（最新）` 的结果项。
- 未知：用户所说“最新”是指按发着信时间本身取最新，还是沿用现有「顧客最新情報」既存链路中的“最新”定义。

- AC-004：`発着信日時（最新）` 可被勾选并成功保存。
- AC-005：勾选 `発着信日時（最新）` 后，进入对应报表查看路径时，画面中能看到 `発着信日時（最新）` 这一列或项目。
- AC-006：当客户存在発着信履歴数据时，`発着信日時（最新）` 展示值与 Unknown 3 最终确认后的“最新”判定口径一致。

## 11. 风险与歧义

- 风险 1：“最新”口径分歧风险。业务语言中的“最新”与现有「顧客最新情報」既存链路中的“最新”可能不是同一件事，若不先拍板，验收时极易产生争议。
- 风险 2：表形式レポート与集計レポート范围风险。若集計レポート也被纳入，本次影响面会明显扩大。
- 风险 3：“順番最后一位”语义风险。仅指系统项目最后，还是包含自定义项目后的全列表最后，视觉结果可能不同。
- 风险 4：无発着信履歴数据时的显示规则风险。若业务未先指定，用户可能会把空白、`-`、`N/A` 视为缺陷。
- 风险 5：若只在部分输出链路支持，可能出现画面与下载结果、对外接口结果不一致的“做一半”风险。
- 风险 6：若业务把 `/admin/cdr/index` 画面看到的顺序理解为“最新”依据，需注意该参考页面的默认显示顺序并不能直接等同于“按発着信時間取最新”。
- 风险 7：权限要求未澄清风险。若后续存在特定角色下不可见 / 不可选的业务要求，而本次提前按全员一致可见处理，容易返工。
- 歧义 1：领导原话是「発信日付」，用户补充是「発着信日時（最新）」。其中同时存在「発信」vs「発着信」以及「日付」vs「日時」两层表述差异，业务含义并未完全等同。
- 歧义 2：用户补充中的 `cdr参照` 与 `* 集計レポート`，目前更像线索而非已冻结业务规则，不能直接视为已确认范围或已确认口径。
