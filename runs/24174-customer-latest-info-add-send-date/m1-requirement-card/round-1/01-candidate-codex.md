# Requirement Card Candidate

## 1. 原始需求保真

领导原话（保真）：

```text
"【① 顧客情報レポートについて】
「顧客最新情報」においても発信日付を表示できるようしたい
※現状は「顧客対応履歴」から全履歴を確認いただき、最新の発信日付を抽出いただく必要がございます。"

ベルシステム２４
```

用户补充原话 / 线索（保真）：

```text
発着信日時（最新）→ 発着信履歴に一番最新→ cdr参照

* 表形式レポート
* 集計レポート
```

## 2. 一句话需求摘要

希望在 `顧客情報レポート` 的 `顧客最新情報` 里支持展示一个新增项目 `発着信日時（最新）`，用于显示该客户的最新发着信时间；但“最新”的判定口径、是否同时覆盖 `集計レポート` 等仍未确认。

## 3. 用户真实目标

- 已确认：减少用户必须去 `顧客対応履歴` 全量查看并手动抽取最新发信日付的操作。
- 已确认：让报表侧可直接看到与客户相关的“最新发着信时间”。
- 推测：希望报表配置页可勾选该项目，并在输出结果中直接展示。
- 未知：用户真实目标是否只针对 `表形式レポート`，还是也要求 `集計レポート` 一并支持。

## 4. 业务对象与操作路径

- 业务对象
  - 已确认：`顧客情報レポート`
  - 已确认：其中的 `顧客最新情報`
  - 已确认：新增项目名为 `発着信日時（最新）`
  - 推测：数据参考 `cdr` / `発着信履歴`
- 操作路径
  - 已确认：目标页面为 `admin/customer_contact_history_reports/add`
  - 已确认：目标位置为 `項目（*必須）`
  - 已确认：该区域当前包含 `出力 / 項目名 / 表示名 / 順番` 相关配置
  - 未知：完整菜单进入路径未确认
  - 未知：是否同一路径同时控制 `表形式レポート` 与 `集計レポート`

## 5. 当前现状

- 已确认：`admin/customer_contact_history_reports/add` 的 `項目（*必須）` 为配置型字段列表，不是页面写死的单个业务字段。
- 已确认：当前 `顧客最新情報` 对应字段列表中没有 `発着信日時（最新）` / `Cdr.start_date`。
- 已确认：当前报表输出依赖已保存的字段配置做展示映射。
- 已确认：`顧客最新情報` 现有代码链路里的“最新”口径，实际是“每客户取最新 `customer_notes.id` 后再关联 cdr”，并不是直接取 `cdr.start_date` 最大值。
- 已确认：`/admin/cdr/index` 页面中的 `発着信時間` 来源是 `cdr.start_date`。
- 已确认：`項目（*必須）` 只直接出现在表形式配置区域；`集計レポート` 使用另一套 X/Y/V 配置方式。
- 推测：如果本次仅追加到当前 `項目（*必須）` 列表，直接影响的是 `表形式レポート`。
- 未知：无 `発着信履歴` 数据时，当前业务预期应该显示空白、`-`、`（空白）` 还是其他值。
- 未知：CSV、API、其他导出是否属于本次验收范围。

## 6. 期望结果

- `api_find_report` 是否需要同步影响。

## 10. 验收标准 Acceptance Criteria

- AC-01：访问 `admin/customer_contact_history_reports/add`，在 `顧客最新情報` 对应的 `項目（*必須）` 中，可手动看到 `発着信日時（最新）`。
- AC-02：`発着信日時（最新）` 在配置界面中可被手动勾选，并可保存到报表配置。
- AC-03：保存后查看对应报表时，可手动确认报表中出现 `発着信日時（最新）` 对应展示。
- AC-04：当同一客户存在多条 `発着信履歴` 时，报表展示值应与“已确认的最新口径”一致。
- AC-05：当同一客户不存在 `発着信履歴` 时，报表展示值应与“已确认的无数据显示规则”一致。
- AC-06：既有项目的显示名、顺番、勾选行为在手动回归时不应异常变化。
- AC-07：若后续确认本次范围包含 `集計レポート`，则需手动验证 `集計レポート` 也可按确认后的规则使用或展示该项。
- AC-08：若后续确认本次范围包含 CSV / API，则需分别手动验证其输出结果与页面规则一致。

## 11. 风险与歧义

- 已确认风险：当前 `顧客最新情報` 代码链路中的“最新”并不是按 `cdr.start_date` 取值；如果业务理解的“最新”是发着信时间最新，则现状口径与需求语义可能冲突。
- 已确认风险：`/admin/cdr/index` 的 `発着信時間` 虽来自 `cdr.start_date`，但页面默认排序更接近 `id DESC`，不能直接等同为严格 `start_date DESC` 口径。
- 已确认风险：`表形式レポート` 与 `集計レポート` 当前配置方式不同，不能自然推定“一个新增项会自动同时适用于两者”。
- 已确认风险：CSV、API 已存在输出链路，若需求范围包含它们，会扩大验收面。
- 已确认风险：无数据时现有不同路径存在空串 / `N/A` / `（空白）` 等不同表现，若不先定业务预期，验收容易反复。
- 歧义：领导原话写的是 `発信日付`，用户补充写的是 `発着信日時（最新）`，两者在“仅发信”还是“发着信”、以及“日付”还是“日時”上存在表述差异。
- 歧义：`順番` 的“最后一位”未说明参照的是哪一组字段边界。
