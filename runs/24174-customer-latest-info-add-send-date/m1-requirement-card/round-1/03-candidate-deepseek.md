# Requirement Card Candidate

## 1. 原始需求保真

```text
"【① 顧客情報レポートについて】
「顧客最新情報」においても発信日付を表示できるようしたい
※現状は「顧客対応履歴」から全履歴を確認いただき、最新の発信日付を抽出いただく必要がございます。"

ベルシステム２４
```

用户补充原话 / 线索：

```text
発着信日時（最新）→ 発着信履歴に一番最新→ cdr参照

* 表形式レポート
* 集計レポート
```

## 2. 一句话需求摘要

在 `admin/customer_contact_history_reports/add` 页面中，为「顧客最新情報」目标报表，在 `項目（*必須）` 区域内追加一个可选字段 `発着信日時（最新）`，用于显示该客户最新一条発着信履歴的 `発着信時間`。

## 3. 用户真实目标

用户（ベルシステム２４）希望不再需要手动从「顧客対応履歴」的全履历中自行确认最新发信日期，而是能够：

- 在「顧客最新情報」报表配置页面中，直接勾选 `発着信日時（最新）` 这个字段
- 报表输出时，直接看到针对每个客户自动取出的最新発着信时间
- 从而减少手工核对步骤，提升业务效率

## 4. 业务对象与操作路径

**业务对象**

- 报表类型：`表形式レポート`（type=1），以及可能涉及的 `集計レポート`（type=2）
- 报表目标：`顧客最新情報`（target=2）
- 追加字段：`発着信日時（最新）`
- 数据来源：`cdr` 表（発着信履歴），参考字段 `発着信時間`（cdr.start_date）
- 关联实体：CustomerContactHistoryReport、CustomerContactHistoryReportField、cdr、customer_notes、customer_contacts

**操作路径**

1. 系统管理员 / 有权限用户访问报表配置页面：
   - 已确认目标URL：`admin/customer_contact_history_reports/add`
   - 完整菜单路径未确认
2. 在新建 / 编辑报表页面，选择 `対象` 为「顧客最新情報」时，`項目（*必須）` 区域出现字段列表
3. 用户勾选 `発着信日時（最新）`，并可设置其「表示名」「順番」
4. 保存报表配置
5. 查看或导出报表时，`発着信日時（最新）` 列展示每个客户的最新発着信时间

## 5. 当前现状

根据代码调查报告已确认：

| 维度 | 现状 |
|------|------|
| `項目（*必須）` 字段源 | `target=2`（顧客最新情報）的系统字段列表中不包含任何 Cdr 相关字段；只有 `target=1`（顧客対応履歴）才包含 `Cdr.start_date`。 |
| 报表输出逻辑 | `target=2` 的报表查询，每个客户是通过 `customer_notes` 最大 `id` 关联最新 note，再 left join cdr 表。并非按 `Cdr.start_date` 降序取最新発着信履歴。 |
| 页面显示 | 配置页面的「項目（*必須）」四列（出力、項目名、表示名、順番）由 `tableSystemFields` 数组渲染，`順番` 默认按数组追加顺序递增。 |
| 无数据展示 | 报表无值时部分分支输出空字符串，部分输出 `N/A` 或 `global.blank`，尚无统一约定。 |
| CSV下载 | 表形式与集計均存在 CSV 下载功能，新增字段会影响对应 CSV 输出列。 |
| 集計レポート | 集計报表配置界面使用 X/Y/V 下拉，不在同一「項目（*必須）」列表内；且 `target=2` 集計 SQL 未 join cdr。 |
| API | 存在 `api_find_report` 接口基于已配置字段输出 CSV。 |
| /admin/cdr/index | 「発着信時間」显示为 `cdr.start_date` 字段，默认排序为 `id DESC`（并非严格按 `start_date DESC`）。 |
| 权限 | privilege.conf 中存在一个未实现对应 action 的 `admin_report_detail_download` 权限项。 |

## 6. 期望结果

1. 当用户在 `admin/customer_contact_history_reports/add`（及编辑页）选择 `対象=顧客最新情報` 并选择报表类型为 `表形式` 时，`項目（*必須）` 中出现 `発着信日時（最新）`，位于「出力 項目名」列和「表示名」列，并且順番默认安排在所有字段的最后一位。
2. 用户可以勾选该字段并保存。
3. 在已保存报表的查看页面（`admin_report_detail_view`）及对应的 DataTables 列表、表形式 CSV 下载中，能展示该字段值。
4. 对于每个客户，该字段显示的值取自该客户発着信履歴中最新的 `発着信時間`（口径需确认，现暂按需求提到的“cdr参照”理解为基于 Cdr 表的最新年月日时）。
5. 已存在的其他字段、显示名、順番、勾选行为不受影响。
6. 未勾选 `発着信日時（最新）` 时，报表输出与现有保持完全一致。

## 7. 明确范围 In Scope

1. **画面追加**  
   - 在 `admin/customer_contact_history_reports/add` 和 `admin/customer_contact_history_reports/edit` 页面，当 `target=2` 且 `type=1`（表形式）时，`項目（*必須）` 的系统字段列表中追加 `発着信日時（最新）`。
2. **表示名与順番**  
   - 表示名（默认）：`発着信日時（最新）`  
   - 順番：追加在现有系统字段及可能的自定义字段之后（最后一位）。
3. **保存与读取**  
   - 用户勾选该字段后，字段信息正常存入 `customer_contact_history_report_fields`。  
   - 报表配置查询 / 编辑时能正确回显勾选状态。
4. **报表输出**  
   - 表形式报表查看页面（DataTables 列表）能显示该列。  
   - 表形式 CSV 下载包含该列，列头使用设定的表示名。
5. **数据取值**  
   - 採用该客户最新的発着信履歴时间（具体口径根据确认后实现，默认期望按 `Cdr.start_date` 降序取第一条）。  
   - 字段名建议为代码方便映射的名称，如 `cdr_latest_start_date`。
6. **回归保护**  
   - 未勾选时报表完全不变。  
   - `target=1` 及其他报表目标不受影响。  
   - 既存报表的原有字段、順番、表示名不变。

## 8. 明确不做 Out of Scope

1. **不涉及 `集計レポート`（type=2）**  
   - 本次不要求集計レポート的 X/Y/V 下拉中追加该字段。  
   - 不修改 `target=2` 集計 SQL 加入 cdr join。  
   - 若后续需要，作为独立需求提出。
2. **不修改 `/admin/cdr/index` 页面**  
   - 该页面仅为业务参考，本次不直接改动其功能或排序逻辑。
3. **不新增 Excel / PDF 导出或邮件配信功能**  
   - 若已有 CSV 之外导出，当前不扩展。
4. **不修改 API 接口的数据结构**  
   - `api_find_report` 暂不保证本次追加字段自动进入其输出（需单独确认后决定）。
5. **不处理无発着信履歴时的特殊显示逻辑设计**  
   - 待 Unknowns 中确认后，由后续工程卡定义。
6. **不修改权限配置**  
   - 不新增或调整 privilege.conf 中未实现的 action 项。

## 9. 需要确认 Unknowns

1. **最新口径定义**  
   「発着信日時（最新）」的“最新”判定口径，是以 `Cdr.start_date` 最大值为准，还是沿用现有「顧客最新情報」中 `customer_notes.id` 最大值的逻辑？  
   领导原话“発着信履歴に一番最新→ cdr参照”偏向于 Cdr，但现有代码实现并不一致，需要最终确认。

2. **是否仅限于「表形式レポート」**  
   用户补充中提及了“表形式レポート”和“集計レポート”，但代码调查显示集計报表字段机制不同。本次要求是否只要求 `表形式レポート` 支持？还是同时要求 `集計レポート` 也能通过某种方式（即使不是同一UI）选择该字段？若只要求表形式，集計明确排除；若需要集計，则影响面显著扩大。

3. **无発着信履歴数据时的显示值**  
   客户没有任何発着信履歴时，该字段应显示什么？  
   - 空（空白）  
   - `-`  
   - `該当なし`  
   - 沿用类似 `customer_notes` 无数据时的表现（当前为 `N/A` 或 `global.blank`）

4. **順番“最后一位”的精确定义**  
   “順番放在最后一位”是指：  
   - 仅在所有系统字段（如 `customer_notes.note_1` 等）之后，但先于可能存在的自定义字段 `CustomerContactExt.field1~`？  
   - 还是在所有字段（含自定义字段）的最后？  
   请确认希望按哪种顺序。

5. **CSV 下载属于验收范围吗？**  
   表形式レポート的 CSV 下载是否需要作为验收的一部分并确保新列出现？目前代码中 CSV 标题使用字段 `name`，数据取 `field_name`，如果新字段按现有模式实现，CSV 会自然跟随。请确认是否需要在验收标准中显式包含 CSV 验证。

6. **`api_find_report` 是否需要同步影响**  
   该 API 当前基于已配置字段输出 CSV。本次是否需要保证当报表 `target=2` 时，新字段同样出现在 API 输出中？如果不作要求，建议后续独立提出。

7. **是否有多语言要求**  
   当前仅有日文表示名要求，但系统支持多语言包。是否考虑其他语言（如中文、英文）的翻译？此次是否仅日文即可？

8. **权限控制**  
   是否需要限制该字段在某种权限下不可见或不可选？目前需求未提及权限，默认不做限制。

9. **报表查看页面中的列宽、排序、筛选**  
   是否需要为新列指定默认列宽、是否可排序、是否可筛选？还是采用系统默认行为即可？

## 10. 验收标准 Acceptance Criteria

* **AC-001**  
  登录系统，访问 `admin/customer_contact_history_reports/add`，选择対象为「顧客最新情報」，报表类型为「表形式」。  
  验证 `項目（*必須）` 区域中出现 `発着信日時（最新）`，位于「出力 項目名」和「表示名」列，且順番在所有可选字段的最后一位。

* **AC-002**  
  勾选 `発着信日時（最新）` 并填写表示名（如需修改），保存报表。  
  验证保存成功，进入编辑页时该字段仍为勾选状态，表示名、順番显示正确。

* **AC-003**  
  在报表查看页面（表形式）中，对于存在発着信履歴的客户，`発着信日時（最新）` 列展示该客户最新的発着信时间（具体时间与手动查询 Cdr 表按业务口径排序的第一条记录相符）。  
  多条履历时展示时间正确为最新一条。

* **AC-004**  
  在表形式レポート的 CSV 下载中，包含 `発着信日時（最新）` 列头（显示名）及对应的数据行。

* **AC-005**  
  在报表查看页面中，取消勾选的报表或创建未勾选 `発着信日時（最新）` 的报表时，报表列表不出现该列，且所有原字段显示仍正常，不受任何影响。

* **AC-006**  
  修改已有旧报表（対象=顧客最新情報、表形式）时，原有勾选字段的表示名、順番、勾选状态全部保持不变，新字段默认为未勾选，位于最后。

* **AC-007**  
  当某客户没有任何発着信履歴数据时，`発着信日時（最新）` 列显示为已确认的显示值（例如空、`-` 等）。此条验证需在 Unknowns 第3项确认后执行。

* **AC-Regression-001**  
  `target=1`（顧客対応履歴）报表的追加/编辑/查看/CSV下载不受任何影响。  
  集計レポート的字段选择、输出不受任何影响。

## 11. 风险与歧义

1. **业务口径偏差风险**  
   现有「顧客最新情報」的“最新”是基于 `customer_notes.id` 最大来关联 note，并非基于 `Cdr.start_date`。如果本次严格按照“cdr参照”取最新発着信時間，则同一客户可能在该报表中出现“最新 note”与“最新発着信”来源不一致的情况（一个取 note 最新，一个取 cdr 最新）。若用户预期是“同一个最新时点的信息”，需要额外协调。

2. **性能风险**  
   查询中增加一个按 `Cdr.start_date DESC` 取最新一条的子查询或窗口函数，在客户和 cdr 数据量较大时可能影响报表生成性能。目前最高效方案可能与现有 `customer_notes.max(id)` 子查询有所不同，需在 Engineering Requirement Card 中设计并评估。

3. **集計レポート遗漏风险**  
   用户补充文本中提到了“集計レポート”，如果最终确认需要支持，当前 In Scope 未包含，将引发返工和额外影响面（需要改动集計 SQL，涉及 group by 和聚合逻辑，可能产生更多歧义）。

4. **順番冲突风险**  
   如果系统字段、自定义字段数量较多，且需求“最后一位”定义不明确，可能导致字段排列不符合用户预期，引发客服投诉或再次调整。

5. **无数据显示不一致风险**  
   代码中不同路径空值处理已不统一（空串、`N/A`、`（空白）` 等）。直接复用某一路径可能与其他字段表现不一致，需要统一策略，否则用户感知混乱。

6. **CSV 输出编码/转义问题**  
   新增日期时间字段在 CSV 输出时可能存在格式转换或特殊字符转义问题，需在开发后加以验证。

7. **API 接口兼容性**  
   若 `api_find_report` 不跟随本次需求变更，外部调用方可能无法获取新字段；反之若变更未通知使用者，可能影响其解析逻辑。需确认影响范围并通知相关方。

8. **只读/安全约束误解风险**  
   未实现的权限项 `admin_report_detail_download` 可能误导认为存在下载权限控制，实际无效。需留意后续清理。

## 12. 给后续 Engineering Requirement Card 的输入提醒

1. **必须优先澄清 Unknowns**  
   特别是“最新口径”（Cdr.start_date 还是 customer_notes.id）、“集計レポート是否纳入范围”、“无数据时的显示值”和“順番的精确位置”。在未确认前，后续 Engineering 不能开始设计实现。

2. **准确定义字段映射标识**  
   新增字段的内部 `field_name`（如 `cdr_latest_start_date`）需要被 mapping logic 识别，并加入表形式 target=2 的查询、DataTables 输出、CSV 输出、保存/读取模型。

3. **查询性能设计**  
   需要给出针对按 `Cdr.start_date` 取最新记录的高效 SQL 方案（如 lateral join、子查询、窗口函数或应用层聚合），并与现有 `customer_notes.max(id)` 逻辑隔离或兼容。

4. **兼容性注意事项**  
   - 保持 `target=1` 和 `target=2` 的字段列表完全独立，改动不影响 target=1。  
   - 仅 type=1 表形式受影响，集計部分代码不得触碰。  
   - 不更改既存字段的默认 ordering 和已保存数据，純粋に追加する。

5. **测试要求**  
   - 覆盖有履历、无履历、多条履历、单条履历的客户数据。  
   - 覆盖新旧报表（含已保存配置）的兼容性。  
   - 覆盖表形式 CSV 下载列序和内容。  
   - 回归测试集計レポート不受影响。

6. **国际化（i18n）**  
   至少提供日文表示名默认值，并建议在语言文件中预留 key，方便日后扩展。

7. **文档与通知**  
   如果确认 `api_find_report` 需要连带变更，需通知集成方；否则在需求中注明不包含。

8. **遗留清理提示**  
   privilege.conf 中的无效权限虽不处理，但可作为 technical debt 记录，后续单独清理。
