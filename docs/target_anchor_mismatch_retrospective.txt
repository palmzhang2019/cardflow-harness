标题:
target 锚点误判复盘：顧客最新情報 与 target=1 / target=2 的对照确认缺失

==================================================
1. 背景
==================================================

任务:
24174-customer-latest-info-add-send-date

原始业务需求围绕:
「顧客最新情報」においても発信日付を表示できるようにしたい

在 M1 / M2 / M3 前半流程中，本任务一直被理解为:

- target=2 / 顧客最新情報
- 在 target=2 的表形式「項目（*必須）」中追加「発着信日時（最新）」
- target=1 / 顧客対応履歴 保持既存行为不变

因此后续实现、修正、Codex Review 都围绕 target=2 展开。

但页面验收阶段，用户实际确认的新目标是:

- target=1 / 顧客対応履歴:
  - 保留既存「発着信時間」
  - 追加「発着信日時（最新）」
  - 两个字段需要同时存在

- target=2 / 顧客最新情報:
  - 不要出现「発着信日時（最新）」

这导致前面已完成的 target=2 方向实现需要反转。

==================================================
2. 问题本质
==================================================

这不是单纯的代码实现错误，也不是某个模型单独写错。

问题本质是:
业务词「顧客最新情報」与代码 / 页面里的 target=1、target=2 之间，没有在进入实现前做最终对照确认。

前面流程默认做了如下映射:

「顧客最新情報」
  -> target=2 / 顧客最新情報
  -> target=2 追加「発着信日時（最新）」

但用户在页面验收时发现，实际想要的是:

target=1 中追加「発着信日時（最新）」
target=2 中不要「発着信日時（最新）」

也就是说:
M1 / M2 阶段冻结了一个后续被证明不符合用户真实意图的 target 锚点。

==================================================
3. 责任链分析
==================================================

3.1 用户输入侧

用户原始输入中包含「顧客最新情報」这个业务词。
该词本身在业务语境中看似明确，但在既存代码 / 页面结构中需要进一步确认它对应哪个 target。

因此用户输入不是“简单错误”，而是存在可被误解的锚点信息。

3.2 陪跑侧

陪跑过程中没有在实现前追问:

这个字段最终应该出现在哪个 target/type 的「項目（*必須）」里？

尤其没有要求用户在实际页面上确认:

- target=1 / 顧客対応履歴 是否需要该字段
- target=2 / 顧客最新情報 是否需要该字段
- 两边是否都要
- 其中一边是否只是参考

因此歧义没有被拦截。

3.3 M1 / M2 冻结侧

M1 / M2 将目标冻结为 target=2。
后续 Engineering Requirement Card、Code Investigation、Phase Plan 都继承了这个前提。

因此错误不在后续某个 patch，而是在较早阶段被固化成了“已确认事实”。

3.4 实现侧

MiniMax / DeepSeek / Codex 后续基本是在执行已经冻结的 target=2 前提。

后续 Review 甚至重点保护:
target=2 新逻辑不能影响 target=1 既存「発着信時間」。

这说明模型链路内部是自洽的，但自洽于错误前提。

3.5 页面验收侧

页面验收阶段，用户实际查看 target=1 / target=2 后，才发现真实意图与已冻结前提不一致。

因此页面验收暴露的是需求锚点问题，而不是单纯 UI 未显示问题。

==================================================
4. 具体出错点
==================================================

最关键出错点:

在 M1 / M2 进入 M3 实现计划之前，没有加入“业务词 ↔ 页面 target/type”的最终确认步骤。

应该确认但没有确认的问题:

字段「発着信日時（最新）」最终应该出现在:

A. target=1 / 顧客対応履歴
B. target=2 / 顧客最新情報
C. 两边都要
D. target=2 只是参考，实际要改 target=1
E. target=1 只是既存参考，实际要改 target=2

如果当时做了这个确认，就不会把 target=2 锁死为实现目标。

==================================================
5. 为什么后续越来越复杂
==================================================

因为 target=1 已经存在:

- field_name/key: Cdr.start_date
- label: 発着信時間

而用户现在希望 target=1 同时保留:

- 発着信時間
- 発着信日時（最新）

这意味着不能简单在 target=1 的 tableSystemFields 中再追加:

Cdr.start_date => 発着信日時（最新）

原因:
PHP 数组 key 不能重复，同一个 key Cdr.start_date 只能存在一个，后者会覆盖前者。

因此新需求需要:

- 为「発着信日時（最新）」设计一个不会覆盖既存 Cdr.start_date 的 alias field_name/key
- 例如 Cdr.start_date_latest
- 但实际取值仍映射到 Cdr.start_date
- 同时确保既存「発着信時間」行为不变

这就是为什么需求反转后，不能只移动一行配置，而需要重新审视 DataTables、CSV、api_find_report 等输出链路。

==================================================
6. 当前已明确的新需求
==================================================

已确认:

target=1 / 顧客対応履歴:

- 保留既存「発着信時間」
- 追加「発着信日時（最新）」
- 两个字段需要同时存在

target=2 / 顧客最新情報:

- 不要出现「発着信日時（最新）」

推测:

- 新字段「発着信日時（最新）」实际取值仍来自 Cdr.start_date
- 新字段无值时仍显示 / 输出 "-"

未知 / 需代码确认:

- 新字段 alias field_name/key 应采用什么命名最安全
- 项目中是否已有类似 alias 字段命名风格
- DataTables 输出链路如何支持 alias -> Cdr.start_date
- 表形式 CSV 如何支持 alias -> Cdr.start_date
- api_find_report 是否也要支持 alias -> Cdr.start_date
- target=2 之前追加的逻辑如何完整撤回

==================================================
7. 对 CardFlow 流程的改进建议
==================================================

7.1 在 M1 Requirement Card 增加 target/type 锚点确认

凡是涉及既存系统中的枚举、target、type、report kind、mode 时，必须新增一项:

Target / Type Anchor Confirmation

模板:

- 业务名称:
- 页面上实际选择项:
- 代码枚举值:
- 最终要出现字段的位置:
- 只是参考的位置:
- 明确不改的位置:

示例:

业务名称:
顧客最新情報

页面 target/type:
target=1 / 顧客対応履歴
target=2 / 顧客最新情報

最终要出现:
target=1 表形式「項目（*必須）」

明确不出现:
target=2 表形式「項目（*必須）」
target=2 集計 X/Y/V

7.2 在 M2 Engineering Requirement Card 增加 anchor confidence

建议增加:

anchor_confidence: confirmed | inferred | unknown

规则:

- 如果 target/type 是用户在页面上实际确认过的，才可写 confirmed
- 如果只是根据业务词推断，必须写 inferred
- 如果代码调查也无法确认，写 unknown

本次错误就是因为:
target=2 实际上是 inferred，但在流程里被当成 confirmed 使用。

7.3 在进入 M3 前增加页面级锚点检查

进入实现计划前，必须让用户确认一句:

请在页面上实际确认:
这个字段最终应该出现在哪个 target/type 的「項目（*必須）」里？

推荐选项:

A. target=1
B. target=2
C. 两边都要
D. target=1 要，target=2 不要
E. target=2 要，target=1 不要
F. 仍不确定，交给代码调查

7.4 对同源字段重复显示增加特殊检查

如果需求要求在同一个 target 中同时显示两个来自同一底层字段的项目，例如:

- 発着信時間
- 発着信日時（最新）

并且它们都可能对应 Cdr.start_date，则必须提前标记:

alias_required: true

并要求代码调查回答:

- 是否允许同一底层字段以两个不同 field_name/key 保存
- alias key 如何命名
- alias 如何映射回真实字段
- 既存字段是否会被覆盖
- 输出链路是否都支持 alias

==================================================
8. 本次教训
==================================================

一句话总结:

本次问题不是代码阶段突然出错，而是需求澄清阶段没有把业务词「顧客最新情報」和实际页面 / 代码 target 值做最终对照确认，导致 target=2 被错误冻结为实现目标。

以后遇到既存系统的 target/type/report 枚举时，不能只根据业务词推断，必须让用户在页面上确认最终字段出现位置。

关键规则:

业务词不能直接等于代码 target。
页面确认优先于语义推断。
inferred target 不能进入实现阶段。
target/type 未确认时，M3 不得开始 implementation prompt。
同一底层字段要显示成两个字段时，必须提前设计 alias key。
